# Pull Request ガイドライン

このプロジェクトのプルリクエストは、以下のルールに従って作成します。

## Description の書き方

### ✅ 書くべきこと（抽象的な背景・意図）
- **Why**: なぜこの変更をするのか
- **What**: 何を実現したいのか
- **How**: どの段階の取り組みか（全体フローの中での位置付け）
- **背景**: 変更に至った経緯や制約

### ❌ 書かないこと（具体的な実装詳細）
- コードを読めばわかる変更内容
- 「xxx ファイルに yyy を追加した」
- 「zzz 関数を新規作成した」
- 「yyy パッケージをインストールした」

### 例：Good PR Description
```markdown
## 背景
Storybook を導入したため、GitHub Pages の公開先をそこに切り替えることで、アニメーションギャラリーとしての利用を実現する段階に入りました。

## 意図
開発環境と公開環境を同じ Storybook で統一することで、一貫性が保たれ、サンプル追加時の手戻りが減ります。

## 関連 Issue
- #2 Storybookの導入
- #6 アニメーションに関するコンテンツの作成

Closes #7
```

## 関連 Issue の紐付け

### 書き方
```markdown
## 関連 Issue
- #2 直接の依存 Issue
- #6 親 Issue
- #7 このプルリクで close される Issue

Closes #7
```

### ポイント
- 関連する全ての Issue をリストアップ
- 「Closes #XXX」で close 対象を明示
- 親 Issue がある場合は必ず記載

## フロー

1. **計画書を作成** → フィードバック・承認
2. **実装・コミット** → 計画書はコミットに積まない
3. **ブランチをプッシュ**
4. **PR を作成** → 上記ルールに従って Description を作成
5. **確認・マージ**

## 計画書の管理

- ファイル名: `feature-xxx.claude.md`
- コミット: **積まない**（開発用ドキュメント）
- 保存: ローカルに残して参考にする
